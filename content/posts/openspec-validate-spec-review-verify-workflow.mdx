## 스펙 주도 개발의 검증에 빈틈이 있었다

OpenSpec 워크플로우는 스펙을 쓰고, 논의하고, 구현하고, 정리하는 흐름이다. 검증이 아예 없었던 건 아니다. apply 단계에서 에이전트가 테스트를 작성하고, tasks.md에 검증 항목도 있었다. 문제는 검증이 apply 안에 뭉뚱그려져 있었다는 거다.

첫째, 포맷 검증이 부족했다. OpenSpec CLI에 `openspec validate`가 있지만, 리포지토리에 맞게 추가한 커스텀 규칙은 검증하지 못했다. 시나리오 ID 중복이나 진입점 파일 누락 같은 건 사람이 리뷰할 때 수동으로 발견했다.

둘째, 스펙 자체를 검토하는 단계가 없었다. 포맷 검증과 구현 사이에 스펙 문서의 논리적 빈틈을 점검하는 과정이 빠져 있었다. apply 에이전트가 구현하면서 알아서 판단했고, 스펙의 빈틈을 자의적으로 해석하거나 스펙이 흔들리는 상황이 생겼다.

셋째, 자기가 짠 코드를 자기가 검증하고 있었다. 구현한 에이전트가 바로 테스트까지 돌린다. 확증 편향과 컨텍스트 오염으로 놓치는 게 생긴다. 테스트는 통과하는데 스펙은 만족하지 않는 상황이 생길 수 있었다.

## 추가한 세 가지 검증 단계

### validate — 형식이 맞는가

validate는 "스펙 문서가 기계적으로 올바른 형식인가?"만 본다. 두 단계로 나뉜다. 먼저 OpenSpec CLI의 `openspec validate --strict`로 기본 구조를 검증하고, 이어서 리포지토리 고유 규칙 6가지를 검사한다.

6가지 규칙은 이렇다.

- 요구사항 분류가 메인 스펙과 일치하는지
- 검증 항목이 존재하는지
- 시나리오 ID가 유일하고 참조가 맞는지
- 시나리오와 검증 항목이 1:1 대응하는지
- 진입점 파일이 실제 존재하는지
- 스펙에 여러 관심사가 섞여 있지 않은지

마지막 항목만 FAIL이 아닌 경고로 처리했다. 기계적으로 딱 자르기 어렵고, 초기에는 false positive가 많을 수 있어서다.

validate는 여기까지다. 스펙 내용이 말이 되는지는 다음 단계에서 본다.

### spec-review — 이 스펙대로 구현하면 흔들리지 않는가

spec-review는 "이 스펙대로 구현하면, 구현 도중에 스펙이 흔들리지 않을까?"만 본다. 스펙이 좋은지가 아니라, 구현에 들어가도 되는지를 판정한다.

5가지 축으로 검증한다.

- 빠진 게 없는지 (완전성)
- 누가 읽어도 같은 뜻인지 (결정론성)
- 글자만 만족하고 의도를 빗겨갈 수 있는지 (우회불가성)
- 다른 관심사가 섞여 있지 않은지 (응집성)
- 수정이라고 분류한 안에 사실상 새 요구사항이 숨어 있지 않은지 (분리성)

걸리는 게 있으면 구현으로 못 넘어간다. 스펙을 다시 논의하거나, 새 change로 분리해야 한다.

구현 도중에 스펙이 흔들리면 어떻게 하는지도 정해뒀다. 바로 구현을 멈추고, 뭐가 흔들리는지만 파악한다. 구현 에이전트가 스펙을 고치지 않는다. 스펙 문제면 논의 단계로 돌아가고, 구현 쪽 오해면 구현만 고치고 계속한다.

### verify — 구현이 스펙을 정말 만족하는가

verify는 구현 후에 독립 에이전트가 검증하는 단계다. 구현한 에이전트와 다른 에이전트가 맡는다.

두 단계로 동작한다.

- **Phase 1**: 테스트가 진짜 결함을 잡을 수 있는지 본다. "이 테스트가 통과하면서도 스펙을 위반하는 구현이 가능한가?"를 따진다. 반례가 없으면 STRONG, 있으면 WEAK, 테스트 자체가 없으면 MISSING이다.
- **Phase 2**: 구현이 스펙 시나리오를 실제로 만족하는지 본다. 구현 의도를 모르는 상태에서 스펙만 보고 코드를 대조한다. PASS, FAIL, PARTIAL로 판정한다.

검증 에이전트는 구현 에이전트가 왜 그렇게 짰는지 모른다. spec.md, tasks.md, git diff, 테스트 코드만 입력으로 받고, design.md나 대화 히스토리는 볼 수 없다. Claude Code에서는 별도 worktree에서, Codex에서는 별도 태스크로 실행해서 컨텍스트를 분리한다.

FAIL이나 NEEDS-WORK 판정이 나오면 verify-report.md가 생성되고, 다음 apply의 입력으로 들어간다. apply > verify > 재작업의 피드백 루프가 돌면서, 문제가 워크플로우 안에서 잡힌다.

## 스펙이 구현을 통제하게 됐다

이전에는 스펙을 작성해도 구현 에이전트가 스펙의 빈틈을 자기 판단으로 메웠다. 스펙이 구현을 안내하는 수준이었지 통제하지는 못했다.

세 단계가 추가되면서 달라진 게 세 가지 있다.

- **스펙의 결함이 구현 전에 드러난다.** spec-review가 "이 스펙대로 구현하면 흔들리는가?"를 사전에 판정한다. 불완전한 스펙이 구현으로 넘어가지 않으니, 에이전트가 알아서 판단할 여지가 줄어든다. 스펙을 더 잘 써야 구현을 시작할 수 있게 된 거다.
- **스펙과 구현의 대응을 추적할 수 있게 됐다.** validate가 시나리오 ID와 검증 항목의 1:1 매핑을 강제한다. 스펙의 어떤 시나리오가 어떤 테스트로 검증되는지 기계적으로 추적된다. "스펙에는 있는데 테스트는 없는" 상황이 구조적으로 방지된다.
- **스펙 준수 여부를 구현자가 아닌 제3자가 판정한다.** verify가 구현 에이전트와 분리된 독립 에이전트로 "코드가 시나리오를 만족하는가?"만 본다. 구현 의도를 모르는 상태에서 스펙과 코드만 대조하니, 스펙에 적힌 것만이 판정 기준이 된다.

스펙 주도 개발에서 스펙은 "구현 전에 쓰는 문서"가 아니라 구현 전·중·후 전 과정에서 판정 기준으로 작동하는 계약이다.

- 구현 전: 스펙 자체의 품질이 검증되어야 구현을 시작할 수 있고 (validate + spec-review)
- 구현 후: 스펙의 시나리오가 독립된 판정 기준으로 작동하며 (verify)
- 전 과정: 스펙과 구현의 대응이 기계적으로 추적된다 (시나리오 ID ↔ 검증 항목 1:1 매핑)

## 검증이 추가된 파이프라인

전체 파이프라인이 이렇게 바뀌었다.

이전: EXPLORE > PROPOSE > DISCUSS > APPLY > ARCHIVE

이후:
```
EXPLORE > PROPOSE > DISCUSS
  > VALIDATE (포맷·추적성 기계 검증)
    > SPEC-REVIEW (스펙 안정성 논리 심사)
      > APPLY (구현)
        > VERIFY (독립 에이전트 검증)
          > ARCHIVE
             ↑ FAIL/NEEDS-WORK > APPLY 재작업
```

validate는 형식 실수를, spec-review는 스펙의 빈틈을, verify는 구현의 빈틈을 잡는다. 세 단계 모두 필수는 아니고, 필요할 때 골라서 쓴다.

## 다음 할 일

이제 필요한 건 이 검증 단계들이 실제로 효과가 있는지 측정하는 거다.

자동으로 수집할 수 있는 건 있다. validate의 FAIL/WARN 횟수, spec-review의 Blocker 수, verify의 판정 결과, 재작업 루프 발생 여부. 각 단계가 실행될 때 리포트 파일이 생기니까 거기서 파싱하면 된다.

하지만 "이 Blocker가 진짜 문제였는가?"나 "archive 이후에 스펙 관련 결함이 나왔는가?"는 사람이 판단해야 한다.

수집은 자동화하고, 판정은 사람이 한다. 사람의 부담을 줄이려면 archive 시점에 "이번 change에서 워크플로우가 잡은 것 중 false positive가 있었나요?" 정도의 질문 하나만 던지는 게 현실적이다.

효과가 없으면 폐기한다. 워크플로우 단계는 비용이다. 잡아내는 게 없는 단계를 유지할 이유는 없다.
