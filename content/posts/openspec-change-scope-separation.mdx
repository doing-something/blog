## 이전 글에서 꺼낸 떡밥

[앞선 글](/blog/agents-md-instruction-enforcement#내-잘못도-있다)에서 "스펙 범위를 넓게 잡아서 작업이 길어졌다"고 짧게 언급했다. LLM이 지시를 안 따르는 원인 중 하나로 컨텍스트가 길어진 걸 꼽았는데, 그 컨텍스트를 길게 만든 게 나였다. 그 얘기를 하려고 한다.

그리고 스펙이 자꾸 바뀌는 게 문제라고도 했었다. 근데 돌아보니 바뀐 게 아니라 내가 계속 추가한 거였다. 체계적으로 일하려고 스펙 주도 개발을 시작했는데, 정작 여러 일을 한 스펙에 섞어서 진행하고 있었다. 그걸 이상하게 여기지도 않았다.

## 실제로 얼마나 심했나

이전 스펙들을 분석해봤다. 하나의 스펙 안에 얼마나 많은 관심사가 섞여 있었는지 정리한 결과다.

| **스펙** | **태스크 수** | **심각도** | **섞인 관심사** |
| --- | --- | --- | --- |
| error-handling | 84 | CRITICAL | 에러페이지 + 알럿 + 인증라우팅 + 로딩 |
| survey-ui-design-system | 68 | HIGH | 토큰정리 + 레이아웃 + 인증 + Flutter브릿지 |
| survey-screen-refactor | 63 | HIGH | 컴포넌트 + 훅추출 + shared UI 구조변경 |
| survey-data-format-update | 48 | HIGH | 데이터모델 + 플로우엔진 + UI + API계약 |
| fix-survey-bugs | 47 | HIGH | 로직버그 + UI이슈 + RangeSlider 리팩토링 |
| mock-api-msw | 52 | MEDIUM | MSW인프라 + 에러분류체계 + 응답포맷팅 |
| simplify-survey-complete-flow | 36 | MEDIUM | 라우트분리 + 캐싱로직 |
| survey-selection-mode-refactor | 20 | LOW | 시맨틱변경 + 리네이밍 |

error-handling 스펙이 태스크 84개다. 에러페이지, 로딩페이지, 알럿, 인증라우팅, 뮤테이션 에러를 전부 한 스펙에 넣었다. 이건 "에러 처리"라는 이름으로 묶었을 뿐, 실제로는 서로 다른 일 네다섯 개를 한꺼번에 한 거다.

## 반복된 패턴 4가지

분석하면서 반복되는 패턴이 보였다.

**아키텍처 레이어 횡단 번들링.** 가장 빈번했다. error-handling이 대표적이다. UI 레이어, 라우팅 레이어, 데이터 레이어를 하나의 스펙에 전부 담았다.

**리팩토링과 기능 추가 혼합.** survey-screen-refactor에서 컴포넌트 구조 변경(리팩토링)과 shared UI 폴더 재구성(인프라)을 함께 진행했다. 독립적으로 리뷰하고 되돌릴 수 있어야 하는 작업을 하나로 묶었다.

**디자인시스템과 아키텍처 혼합.** survey-ui-design-system에서 토큰 정리, SurveyLayout 재설계, Flutter 브릿지 계약 변경을 한 스펙에 68개 태스크로 넣었다. 디자인시스템 작업과 레이아웃 아키텍처 작업은 다른 일이다.

**버그수정이라는 이름의 리팩토링.** fix-survey-bugs에서 실제 버그 수정 외에 RangeSlider 포인터 이벤트 시스템을 전면 교체했다. 이건 fix가 아니라 refactor 스펙으로 분리했어야 했다.

## 어쩌다 이렇게 됐나

처음부터 비대하게 설계한 게 아니다. 완성된 스펙 위에 관련 작업을 계속 쌓은 결과다. 흐름은 이랬다.

티켓 작업 시작 → 스펙 작성 → 태스크 실행 → 완료 가능한 상태 → "이것도 같이 배포하고 싶다" → 스펙에 태스크 추가 → 추가분 완료 → 또 추가 → 한꺼번에 배포.

AI로 실행이 빨라지면서 이 루프가 가속됐다. 원래라면 "이건 다음에 하자"라고 끊었을 작업을 "금방 되니까 같이 하자"로 바꿨다. 기존에 지키던 "한 번에 하나"라는 원칙을 AI의 속도가 무너뜨렸다. 정확히는, 빠르니까 괜찮다고 스스로 허용한 거다.

AI를 쓰더라도 기존의 작업 원칙을 무너뜨리면 안 된다. 실행 속도가 올바른 원칙을 너무 쉽게 어기게 만드는 위험이 있다.

## 체계적이라는 착각

태스크를 만들어 순서대로 작업하고, 시나리오대로 검증하면서 진행했다. 스펙이 있고, 태스크가 있고, 검증 절차가 있으니까 시스템적으로 일하고 있다고 생각했다.

근데 실제로는 혼란을 만들고 있었다. 태스크 하나하나는 순서대로 잘 했는데, 그 태스크들이 애초에 한 스펙에 있으면 안 되는 것들이었다.

나한테 이게 일종의 hallucination이었다. 시스템의 형식을 갖추고 있으니까 시스템적으로 일하고 있다고 믿었다.

## 스펙 하나에 닫힌 문제 하나

실제로 필요했던 건 이런 흐름이다.

같은 브랜치에서 작업하더라도: 스펙1(핵심 기능) → 완료 → 스펙2(발견된 리팩토링) → 완료 → 스펙3(추가 개선) → 완료 → 배포.

스펙 하나에 닫힌 문제 하나만 담는다. 작업하다 추가 작업이 보이면 현재 스펙에 넣지 않고 별도 스펙으로 분리한다. 배포 단위는 여러 스펙을 묶을 수 있지만, 스펙 자체의 범위는 섞지 않는다.

이건 새로운 원칙이 아니다. 원래 알고 있던 거다. PR 하나에 하나의 관심사, 커밋 하나에 하나의 변경. 스펙 주도 개발에서도 똑같다. AI를 쓴다고 이 원칙이 바뀌는 건 아니다.

## OpenSpec의 change 분리 원칙

사실 이건 내가 OpenSpec을 잘못 쓴 거다. 문서를 제대로 읽고 따랐으면 피할 수 있었다.

기존에는 새 도구를 도입할 때 공식 문서를 읽어보고 시작했다. 이번에는 먼저 적용해보면서 AI의 도움을 받아 고치는 걸로 시작했고, 그래서 이런 일이 생겼다.

[OpenSpec 문서](https://github.com/Fission-AI/OpenSpec/blob/main/docs/concepts.md)에는 change를 분리하는 기준이 명확하게 나와 있다.

새 change를 만들어야 하는 경우:
- 의도 자체가 바뀌었을 때 — "다크모드 추가"가 "커스텀 테마 시스템 구축"이 된 경우
- 범위가 폭발했을 때 — 원래 proposal을 알아볼 수 없을 정도로 커진 경우
- 원래 change를 완료할 수 있을 때 — 새 작업이 개선이 아니라 독립된 작업인 경우

결정 휴리스틱도 있다.

```
                    ┌─────────────────────────────────────┐
                    │     Is this the same work?          │
                    └──────────────┬──────────────────────┘
                                   │
                ┌──────────────────┼──────────────────┐
                │                  │                  │
                ▼                  ▼                  ▼
         Same intent?      >50% overlap?      Can original
         Same problem?     Same scope?        be "done" without
                │                  │          these changes?
                │                  │                  │
      ┌────────┴────────┐  ┌──────┴──────┐   ┌───────┴───────┐
      │                 │  │             │   │               │
     YES               NO YES           NO  NO              YES
      │                 │  │             │   │               │
      ▼                 ▼  ▼             ▼   ▼               ▼
   UPDATE            NEW  UPDATE       NEW  UPDATE          NEW
```

이 기준대로라면 내 change들은 대부분 NEW였어야 했다. 원래 change를 "완료"로 표시할 수 있는 상태에서 추가 작업을 쌓았으니까. 도구가 이미 이 흐름을 지원하고 있었는데, 나는 그걸 무시했다.

change를 쪼개면 전체를 어떻게 관리하느냐는 질문이 생긴다. 앞선 글에서 "스펙을 메인 스펙과 서브 스펙으로 나눠서 처리하는 방안이 필요하다"고 했는데, OpenSpec에 이미 있었다.

change 간 의존성을 proposal의 Dependencies 섹션으로 명시할 수 있다. "Builds on: add-basic-auth"처럼 순서를 잡고, 각 change를 개별로 진행한 뒤 bulk-archive로 한번에 정리한다. 쪼개되 흩어지지 않는 구조다.

[OpenSpec](https://github.com/Fission-AI/OpenSpec/blob/main/docs/concepts.md)은 네 가지 원칙 위에 만들어졌다. 유연함(fluid not rigid), 반복적(iterative not waterfall), 간결함(easy not complex), 브라운필드 우선(brownfield-first). change를 작게 쪼개는 것도 유연함의 맥락이다. 엄격한 규칙이 아니라 흐름을 가볍게 유지하는 방식이다.

OpenSpec을 선택한 건 이 철학에 공감해서다. 단순하고 유연한 도구로 도구 사용법이 아닌 스펙 주도 개발 자체에 집중하고 싶었다. 근데 정작 제대로 이해하지 못하고 썼다. 그게 이번 불찰이다.
