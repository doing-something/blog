OpenSpec 관련 마지막 글을 작성한 2026년 4월 19일 이후부터 개선한 것들 중간 정리한다.

## 기존 스펙 주도 개발 방식 

OpenSpec의 핵심은 검증 가능한 행동 계약을 문서화하는 것이다.
BDD의 구문적 패턴을 차용하여 행동중심 명세서를 작성한다.
이걸 시나리오 방식으로 하는데, 예를 들어 이렇다.

```
Scenario: 사용자가 잘못된 입력을 제출하는 경우
WHEN 사용자가 필수 값을 비운다
THEN 저장이 차단된다
AND 오류 메시지가 표시된다
```

요구사항을 구조화할 수 있지만, 이것만으로 제품 개발을 하기에는 부족하다.

- 작성된 스펙이 구현 가능한 수준인지 애매할 수 있다
- 구현자가 스펙 의도와 다르게 해석할 수 있다
- 작업 목록은 있지만 무엇을 검증해야 하는지 불명확할 수 있다
- 테스트를 추가해도 실제 요구사항을 검증하지 않을 수 있다
- 코드 리뷰가 스펙 의도보다 코드 스타일 위주로 흐를 수 있다

때문에 작동을 멈추고 중간에 개입하는 일이 매우 잦아진다.
이런 문제를 줄이기 위해 OpenSpec 위에 개발 하네스를 붙였다.
스펙을 작성하는 것에서 끝나지 않고, 다음 흐름을 안정화하는 게 목적이다.

스펙 작성 순서
1. 구현 가능성 검토
2. 스펙 기준 구현
3. 구현 리뷰
4. 스펙 기준 검증
5. 완료된 스펙 반영
6. 반복 문제 회고

즉, OpenSpec을 문서 형식으로만 쓰지 않고, agent가 구현과 검증을 수행할 때 기준으로 삼을 수 있는 실행 계약으로 만드는 것이 핵심이다.

## 개선 사항

### 1. opsx-kit으로 관리하기

여러 기기의 여러 프로젝트에서 공통으로 사용하기 위해 private GitHub 저장소를 만들었다.
써보면서 불편한 점 고쳐서 버전 업그레이드 하고 있다.
스펙이 실제 구현과 검증까지 제대로 이어지도록 워크플로우를 보강해왔다.

### 2. 스펙 문서만으로 구현 가능성을 판단하게 했다

기존에도 구현 전에 스펙을 리뷰하는 단계는 있었는데, 이 단계를 더 엄격하게 만들었다.
스펙 리뷰는 이미 작성된 코드나 이전 대화에서 나온 설명을 참고하지 않고, 스펙 문서에 실제로 적힌 내용만 기준으로 판단하게 했다.

예를 들어 스펙에 이렇게만 적혀 있으면 부족하다.
```
권한 정책을 반영한다
```

이전 대화에서 “관리자가 아니면 저장 버튼을 숨긴다”고 말했더라도, 그 내용이 스펙에 없으면 구현 준비가 된 것으로 보지 않는다.
```
WHEN 관리자 권한이 없는 사용자가 설정 화면에 접근한다
THEN 저장 버튼이 표시되지 않는다
```

### 3. 시나리오-작업-검증 연결을 더 엄격하게 다루게 했다

작업이 어떤 시나리오를 구현하는지, 검증 항목이 어떤 시나리오를 확인하는지 문서에 명확히 드러나야 한다.
이 연결이 불명확하면 구현이나 검증이 충분하다고 보지 않는다.

예를 들어 시나리오가 이렇게 있다.
```
WHEN 권한이 없는 사용자가 설정 화면에 접근한다
THEN 저장 버튼이 표시되지 않는다
```

좋지 않은 작업 목록은 이런 형태다.

- 권한 처리 수정
- 테스트 추가

이렇게 쓰면 무엇을 구현하고 무엇을 확인하는지 알기 어렵다.
더 나은 형태는 다음과 같다.

- 권한이 없는 사용자가 설정 화면에 접근할 때 저장 버튼이 표시되지 않도록 수정
- 권한이 없는 사용자 상태에서 설정 화면을 열었을 때 저장 버튼이 없는지 확인

이렇게 쓰면 작업과 검증이 같은 시나리오를 다룬다는 것을 알 수 있다.

이런 연결 구조는 원래도 있었지만, 너무 일반적인 문장으로도 통과될 수 있었다.
개선 이후에는 그 연결이 구현자가 바로 따라갈 수 있고, 검증자가 확인할 수 있는 수준이어야 한다는 기준을 강화했다.

### 4. 한 번에 처리하는 구현 범위를 더 작게 고정했다

agent가 여러 작업을 한 번에 묶어서 처리하지 않도록, 현재 구현할 작업 범위를 더 작게 고정했다.
예를 들어 작업 목록이 이렇게 있다고 하자.
(여기서 작업 목록은 OpenSpec change에 포함된 tasks.md를 의미한다. tasks.md에는 해당 변경을 구현하기 위해 수행할 작업들이 체크리스트 형태로 정리된다.)

```
- [ ] 권한 없는 사용자에게 저장 버튼 숨기기
- [ ] 저장 API에서 권한 없는 요청 차단하기
- [ ] 권한 오류 메시지 정리하기
```

이전에는 agent가 세 항목을 한 번에 구현할 수 있었다.
opsx-kit에서는 먼저 하나만 처리한다.

```
이번 구현 범위:
- 권한 없는 사용자에게 저장 버튼 숨기기
```

이 작업이 끝나면 검증하고, 필요한 경우 커밋한 뒤 다음 항목으로 넘어간다.
목적은 구현 범위를 작게 유지해서 어떤 코드 변경이 어떤 작업에서 나온 것인지 분명하게 남기는 것이다.

### 5. 구현 전에 현재 작업과 관련된 스펙 근거를 확인하게 했다

구현을 시작하기 전에 현재 tasks.md 항목이 어떤 스펙 근거와 연결되는지 확인하게 했다.
예를 들어 현재 작업이 다음과 같다고 하자.

```
- [ ] 권한 없는 사용자에게 저장 버튼 숨기기
```

구현 전에 다음을 확인한다.

- 이 작업은 어떤 요구사항에서 나온 것인가
- 이 작업과 관련된 시나리오는 무엇인가
- 설계 문서에 관련 결정이 있는가
- 이 작업은 무엇으로 검증할 것인가

이 정보가 문서에서 확인되지 않으면 바로 구현하지 않는다.
스펙을 보완하거나 확인이 필요한 상태로 본다.

### 6. 검증 항목을 “테스트 추가”보다 구체적으로 쓰게 했다

기존에도 tasks.md에는 검증 항목이 있었다.
하지만 검증 항목이 다음처럼 쓰이면 충분하지 않다.

```
- [ ] 테스트 추가
- [ ] 권한 처리 검증
```

이렇게 쓰면 무엇을 어떤 방식으로 확인해야 하는지 알기 어렵다.
opsx-kit에서는 검증 항목이 다음 내용을 포함하게 했다.

- 무엇을 확인하는가
- 어떤 방식으로 확인하는가
- 성공 조건은 무엇인가

예를 들어 더 나은 검증 항목은 다음과 같다.

```
- [ ] 권한 없는 사용자로 설정 화면을 열었을 때 저장 버튼이 표시되지 않는지 확인한다.
```

또는 자동 테스트가 적절하지 않다면 이렇게 쓸 수 있다.

```
- [ ] 권한 없는 사용자 계정으로 실제 화면에 접근해 저장 버튼이 보이지 않는지 수동 확인한다.
```

여기서 중요한 것은 테스트를 반드시 추가하는 것이 아니다.
해당 요구사항이 충족됐다고 판단할 근거를 남기는 것이다.

### 7. 검증 범위를 구체적으로 쓰게 했다

기존에도 검증 항목은 있었지만, 범위가 넓거나 애매하게 적힐 수 있었다.
예를 들어 다음 문장은 부족하다.

```
- [ ] 권한 처리 테스트
```

검증 항목에 확인할 동작과 성공 조건을 더 구체적으로 쓰게 했다.

```
- [ ] 권한 없는 사용자로 설정 화면을 열었을 때 저장 버튼이 표시되지 않는지 확인한다.
```

또는 다음처럼 쓸 수 있다.

```
- [ ] 권한 없는 사용자가 저장 API를 호출했을 때 요청이 거부되는지 확인한다.
```

목적은 agent가 너무 좁은 부분만 확인하고 검증을 끝내거나, 요구사항과 다른 부분을 테스트하는 일을 줄이는 것이다.

### 8. 모든 검증을 자동 테스트로 처리하지 않게 했다

초기에는 검증 항목을 자동 테스트 중심으로 쓰는 경향이 있었다.
자동 테스트는 중요하지만, 모든 경우에 가장 좋은 검증 방식은 아니다.
예를 들어 다음과 같은 동작은 자동 테스트로 만들기 어렵거나, 테스트를 만들더라도 실제 동작 확인과 거리가 멀어질 수 있다.

- 브라우저나 디바이스 상태에 의존하는 동작
- 카메라, 타이머, 애니메이션 프레임과 관련된 동작
- 복잡한 UI 상호작용
- 외부 환경이나 실제 운영 설정에 묶인 동작

이런 경우 억지로 자동 테스트를 만들면 mock이 많아지고, 실제로 확인해야 할 동작은 제대로 확인하지 못할 수 있다.
상황에 따라 다음을 검증 근거로 사용할 수 있게 했다.

- 자동 테스트
- 정적 확인
- 수동 확인
- 기존 테스트
- 코드 흐름 확인
- 지금 자동화하지 않는 이유와 후속 확인 방법

목적은 자동 테스트 수를 늘리는 것이 아니라, 요구사항이 충족됐다는 근거를 적절한 방식으로 남기는 것이다.

### 9. 구현 리뷰와 스펙 검증을 구분하게 했다

기존에도 구현 후 리뷰와 검증은 있었지만, 무엇을 보는 단계인지 섞이기 쉬웠다.
opsx-kit에서는 구현 리뷰와 스펙 검증의 질문을 구분했다.
구현 리뷰는 코드 변경 자체를 본다. 예를 들어 다음을 확인한다.

- 코드가 불필요하게 복잡하지 않은가
- 기존 구조를 깨지 않았는가
- 예외 상황이 빠지지 않았는가
- 상태나 비동기 처리가 안전한가
- 에러 처리가 적절한가
- 보안이나 성능 문제가 없는가
- 테스트가 의미 있는 동작을 확인하는가

스펙 검증은 구현 결과가 스펙을 만족하는지 본다.

- 스펙에 적힌 시나리오가 구현됐는가
- 검증 항목이 실제로 실행됐는가
- 검증 근거가 충분한가
- 구현 중 발견한 새로운 요구사항이나 검증 공백이 있는가

즉, 구현 리뷰는 “코드가 괜찮은가”를 보고, 스펙 검증은 “스펙 의도를 만족했는가”를 본다.
이 둘을 나눈 이유는 코드가 좋아 보여도 스펙 요구사항을 빠뜨릴 수 있고, 반대로 스펙은 만족하지만 구현 품질에 문제가 있을 수 있기 때문이다.

### 10. 변경 내용에 따라 확인할 것을 다르게 정했다

스펙 기반 워크플로우는 품질을 높이지만, 모든 변경에 같은 절차를 적용하면 너무 무거워진다.
변경 성격에 따라 가볍게 처리할지, 더 깊게 검토할지 구분하게 개선했다.

문서 문구만 수정하는 경우에는 관련 문서, 반복되는 사본, 형식 정도를 확인하면 충분하다.
반대로 권한, 저장, 데이터 처리, 보안, 사용자 동작처럼 실제 실행 결과가 바뀌는 경우에는 스펙, 관련 코드 흐름, 예외 상황, 검증 근거까지 확인해야 한다.

목적은 단순한 변경에 불필요하게 많은 절차를 적용하지 않고, 실제 동작이 바뀌는 변경에서는 필요한 확인을 빠뜨리지 않게 하는 것이다.

### 11. 반복되는 문제를 워크플로우 개선으로 되돌리게 했다

opsx-kit을 실제 프로젝트에서 쓰다 보면 같은 문제가 반복해서 나타난다.
예를 들면 다음과 같다.

- 스펙 리뷰에서 같은 종류의 누락이 반복됨
- 검증 항목이 계속 모호하게 작성됨
- 테스트는 추가됐지만 실제 요구사항 검증은 약함
- agent가 스펙에 없는 내용을 추측해서 구현함
- 특정 단계가 계속 오래 걸림

이런 반복 문제를 공통 워크플로우 개선 후보로 보게 했다.
예를 들어 검증 항목이 계속 모호하다는 문제가 반복되면, 개별 문서만 고치는 것이 아니라 검증 항목 작성 규칙 자체를 강화한다.

- 반복 문제: 검증 항목이 "테스트 추가"처럼 모호하게 작성됨
- 워크플로우 개선: 검증 항목에 무엇을 확인하는지, 어떤 방식으로 확인하는지, 성공 조건을 쓰게 함

또 다른 예시는 다음과 같다.

- 반복 문제: 단순한 문서 변경에도 agent가 너무 많은 문서를 읽고 오래 걸림
- 워크플로우 개선: 문서 변경처럼 단순한 작업은 필요한 문서와 확인 항목만 보게 함

목적은 같은 실수를 프로젝트마다 다시 해결하지 않고, 공통 워크플로우 자체를 개선해서 다음 변경부터 같은 문제가 덜 발생하게 하는 것이다.

### 12. 작업 성격에 따라 읽을 기준 문서를 정리했다

워크플로우가 커지면서 참고해야 할 문서가 많아졌다. 
예를 들면 다음과 같다.

- 스펙 작성 규칙
- 검증 항목 작성 규칙
- 테스트 품질 기준
- 구현 원칙
- 보안 기준
- 프로젝트별 규칙
- 리뷰 기준

기존에는 agent가 필요한 문서를 놓치거나, 반대로 모든 문서를 매번 읽어서 작업이 무거워질 수 있었다.
opsx-kit에서는 언제 어떤 문서를 읽어야 하는지 정리했다.
예를 들어 문서만 수정하는 작업이면 다음 정도를 읽으면 된다.

- 관련 문서
- 문서 작성 규칙
- 같이 수정해야 하는 반복 문서

테스트를 새로 작성하는 작업이면 추가로 다음을 읽어야 한다.

- 테스트 품질 기준
- 해당 기능의 시나리오
- 검증 항목

모든 기준을 처음부터 prompt에 넣지 않고, 어떤 상황에서 어떤 문서를 읽을지를 분리했다.
그리고 각 워크플로우 지침에 이 경우에는 이 문서를 읽어라를 넣었다.
apply를 시작할 때 helper가 현재 tasks.md의 다음 작업을 보고, 관련 정보를 모아준다.
agent가 매번 처음부터 무슨 문서를 읽어야 하지?를 다시 판단하지 않도록, 작업 단위로 필요한 문서 후보를 구조화해서 넘긴다.
목적은 필요한 기준은 놓치지 않고, 필요 없는 문서를 매번 읽지 않게 하는 것이다.

### 13. 보안과 공통 엔지니어링 기준을 추가했다

기존에는 구현이나 리뷰를 할 때 참고할 공통 기준이 부족했다.
그래서 agent가 프로젝트마다 다른 기준으로 판단하거나, 이전 대화에서 나온 기준에 의존할 수 있었다.
opsx-kit에서는 여러 프로젝트에서 공통으로 쓸 수 있는 기본 기준을 문서로 정리했다.
특히 보안 관련 변경은 별도 기준을 두었다.
목적은 구현과 리뷰에서 매번 기준을 새로 만들지 않고, 여러 프로젝트에서 반복되는 기본 판단 기준을 공유하는 것이다.

### 14. 규칙 위반을 자동으로 잡게 했다

사람이 매번 확인하지 않아도 되는 규칙은 자동 검사로 잡게 했다.
예를 들어 다음 같은 경우다.

- 시나리오 ID가 중복됨
- 검증 섹션 이름이 잘못됨
- 요구사항 문장이 너무 모호함
- 검증 항목이 시나리오와 연결되지 않음
- feedback 형식이 잘못됨

이런 문제는 형식이 정해져 있으므로 도구가 먼저 잡을 수 있다.
목적은 스펙 리뷰에서 매번 같은 형식 오류를 사람이 반복해서 찾지 않게 하는 것이다.

### 15. 실행 기록을 남기고 회고에 쓰게 했다

각 작업 흐름에서 실행 기록을 남기게 했다.
예를 들면 다음을 기록한다.

- 스펙 리뷰가 얼마나 걸렸는가
- 구현 단계가 얼마나 걸렸는가
- 검증 단계가 얼마나 걸렸는가
- 어떤 단계에서 다시 작업했는가
- 검증 근거가 부족하다는 지적이 있었는가
- 같은 종류의 문제가 반복됐는가

이 기록은 기능 동작을 판단하는 근거가 아니다.
개발 워크플로우 자체를 개선하기 위한 자료다.
목적은 같은 불편을 매번 임시로 해결하지 않고, 실행 기록을 바탕으로 공통 워크플로우를 고치는 것이다.

## 마무리

요즘 모델이 좋아져서 알아서 잘하는데 "굳이 이렇게까지 해야 하나?"라는 생각이 들 수도 있다.  
실제로 이 방식은 다소 번거롭고 시간도 더 걸리기 때문에 호불호가 갈리는 것이 당연하다.  
하지만 이 방식으로 개선한 이후에는 내가 워크플로우를 따라가며 검토만 하고, 직접 개입하는 경우는 크게 줄었다.  
내가 개입해야 하는 상황이 발생하더라도, 대부분은 스펙을 작성하는 과정에서 내가 문제를 정의하거나 해결 방식을 설계할 때 놓친 부분이 있었기 때문이었다.

목적은 더 중요한 문제에 시간과 노력을 쓰기 위해서다. 
반복적인 작업에 개입하는 시간을 줄이고, 문제를 정의하고, 여러 관점에서 검토하며, 적절한 해결 방안을 설계하는 과정에 더 집중하고 싶다. 앞으로의 개선할 때는 이 부분에서 더 잘하는 방법을 고민하려고 한다.
