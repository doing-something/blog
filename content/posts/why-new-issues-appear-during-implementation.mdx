어떤 사람들은 여행 계획이나 집안 청소 같은 일상적인 일들을 큰 노력 없이도 정리해서 해낸다.
반면 나는 정리하고 움직이는 데 늘 의식적인 노력이 필요한 편이라, 업무에서는 특히 더 시스템을 만들고 그 안에서 일하려고 노력해왔다.

OpenSpec의 흐름은 사람이 문제를 풀 때 머릿속에서 밟는 순서와 닮아 있다. 
왜 이 일을 하는지 정리하고, 무엇을 만들지 정하고, 어떻게 만들지 설계하고, 어떤 단계로 나눌지 정한 뒤, 마지막으로 구현에 들어간다.
OpenSpec은 이 자연스러운 사고 흐름을 명시적인 워크플로우로 바꿔준다.
그래서 구조와 시스템이 있어야 더 잘 움직이는 나 같은 사람에게 특히 효과적이다.

## 몇 달 써보고 느낀 것들

몇 달 동안 이 방식으로 일해보니, 업무 효율을 높이기 위해 필요한 것 중 간단하게 적용할 수 있는 건 다음과 같다.

### AI 생성 코드의 일관성 확보 필요

코드 리뷰 전에 lint와 타입 검사를 거쳐 리뷰 노이즈를 사전에 줄이고, PR에서는 로직 검증에만 집중하듯,
AI가 생성하는 코드도 프로젝트의 가이드라인을 반영해 리뷰 단계에서 발생하는 컨벤션 노이즈를 최소화할 필요가 있다.
다만 이를 AGENTS.md나 CLAUDE.md 같은 문서에만 의존해 해결하기는 어렵다. 
규칙이 많아질수록 문서는 비대해지고, 실제 생성 코드가 이를 일관되게 따르지 않는 경우도 생긴다. 
따라서 사전 지시와 함께 커밋 전 검증 도구를 두어 규칙 준수 여부를 확인하고, 가능한 경우 자동 수정까지 수행하는 방식이 필요하다.

### Atomic Commit

스펙에 대한 이해가 전제된 상태라면, 커밋 단위가 작을수록 AI가 코드를 작성하는 중간에도 변경 내용을 리뷰하기 쉽다.

### 스펙 문서를 다루기 좋은 작업 환경

스펙 주도 개발에서는 스펙 문서를 읽고, 작성하고, 수정하는 일이 개발 과정의 핵심이 된다.
스펙 문서를 쉽게 읽고, 작성하고, 수정할 수 있어야 한다. 그래야 문서와 구현 사이를 오가며 작업하는 흐름이 끊기지 않는다.

이것들은 비교적 쉽게 적용할 수 있는 것들인데, 하나 아직 어려운 게 있다. 변경 단위를 어디서 나눠야 하느냐다.

## 왜 스펙을 꼼꼼하게 써도 구현 단계에서 추가 변경이 생기는가?

변경을 확정하는 것은 문서로 문제를 정리하고 결정하는 일이고, 구현은 그 내용을 실제 코드와 시스템 안에서 실현 가능한지 확인하는 일이다. 
둘 다 같은 변경을 다루지만, 구현에 들어가면 문서에서는 보이지 않던 제약과 빈틈이 더 구체적으로 드러난다.

### 왜 구현 단계에서만 보이는가?

1. 문서에서는 할 일을 정하면 되지만, 구현에서는 하는 방법까지 끝까지 정해야 한다.
   문서에는 "뒤로가기를 처리한다"라고 쓸 수 있다. 하지만 실제 코드에서는 누가 이벤트를 받고, 상태는 어디서 읽고, 언제 정리할지까지 다 정해야 한다.
2. 문서에서는 따로 보이던 문제가 구현할 때는 한곳에서 부딪힌다.
   상태 구조, 훅 동작, 외부 API 같은 건 문서에서는 나눠서 볼 수 있다. 그런데 코드를 짜기 시작하면 이게 한 함수나 한 흐름 안에서 같이 엮인다.
3. 구현에 들어가야 정말로 검증 가능한 단위인지 알 수 있다.
   요구사항이 맞아 보이는 것과, 실제로 테스트할 수 있게 잘 나뉘어 있는 것은 다르다. 구현해보면 그제서야 "이걸 어떻게 테스트하지?"가 분명해진다.
4. 사람은 문서를 읽을 때 빈칸을 자연스럽게 메운다.
   문서를 읽을 때는 중간이 조금 비어 있어도 "이 정도면 되겠지" 하고 넘어간다. 하지만 코드는 그런 식으로 넘어갈 수 없다. 빠진 부분마다 실제 결정을 요구한다.
5. 구현 중 새로 생긴 문제처럼 보여도, 사실은 원래 있던 미정 사항이 드러난 경우가 많다.
   구현하다가 보이는 "구멍"은 꼭 요구사항이 늘어난 게 아니다. 처음부터 있었지만 문서에서 아직 자세히 나뉘지 않았던 문제가 구현하면서 드러난 것일 때가 많다.

## 그래서 변경 단위는 기능 크기보다 질문 기준으로 나눠야 한다.

변경은 작은 기능 단위보다 하나의 불확실성을 닫는 단위로 나누는 게 더 좋다.
문서에서는 하나로 보였던 변경이, 실제 구현에 들어가면 여러 갈래의 결정을 요구하게 되는 경우가 있다.

### 대표 사례: survey-real-api는 처음부터 나눠졌어야 했다

survey-real-api 변경은 이름만 보면 하나의 작업처럼 보인다. 하지만 실제로는 성격이 다른 질문이 여러 개 섞여 있었다. 그래서 처음부터 하나의 변경으로 잡기보다, 아래처럼 나누는 편이 더 맞았다.

**(1) 실 API 연결 바꾸기**  

이 작업은 가짜 API 연결을 실제 서버 연결로 바꾸는 일만 다룬다.
- API 주소 바꾸기
- GET /surveys와 POST /surveys/answers에 연결하기
- SDK 생성 붙이기
- MSW 엔드포인트 맞추기  

질문: "설문 API 호출 경로를 실제 서버 기준으로 어떻게 교체할까?"

**(2) 설문 조회 응답 규칙 정하기**  

이 변경은 서버가 내려주는 설문 응답을 프론트가 어떤 규칙으로 읽을지를 다룬다.
- memberSubscriptionId를 어디서 받을지
- questions를 어떤 모양으로 파싱할지
- exerciseAssignmentAnswers를 조회 응답에서 어떻게 읽을지
- selectionMode 같은 필드를 어떻게 해석할지

질문: "GET /surveys 응답을 프론트가 어떤 데이터 계약으로 파싱하고 사용할까?"

**(3) 설문 답변 저장 형식 정하기**  

이 변경은 답변을 서버에 어떤 형식으로 보내고, 나중에 어떤 형식으로 다시 받을지를 다룬다.
- answers를 문자열로 보낼지
- 배열 그대로 저장할지
- 조회할 때 exerciseAssignmentAnswers와 어떤 관계를 가질지

질문: "POST와 GET이 공유하는 답변 저장 형식을 무엇으로 정할까?"

**결국 survey-real-api는 하나의 작업이 아니었다. 실제로는 아래 3가지가 섞여 있었다.**
- 서버 연결 방식
- 조회 응답 해석 규칙
- 답변 저장 형식

이 세 가지는 서로 다른 문제다.
하나를 정한다고 다른 하나가 자동으로 같이 정해지지 않는다.
그래서 처음부터 "실 API 전환"이라는 큰 이름으로 묶기보다, 각 변경이 어떤 질문을 닫는지를 기준으로 나누는 편이 더 좋았다.

## 결론

구현 단계에서 새 이슈를 완전히 없애는 것은 현실적인 목표가 아니다.
더 중요한 것은 새 질문이 생겼을 때 그것이 기존 스펙 안에서 이미 답이 있는 구현 문제인지, 현재 변경 안에서 결정을 보강하면 되는 설계 문제인지, 아니면 범위를 넘어서므로 별도의 변경으로 분리해야 하는 문제인지를 바로 분류할 수 있게 만드는 것이다.
그래서 변경의 경계는 기능명이 아니라 이번 변경이 닫아야 하는 질문을 기준으로 잡아야 하고, spec review는 모든 이슈를 제거하는 단계가 아니라 구현 전에 남아 있는 질문과 기준선을 분명히 해두는 단계가 되어야 한다.
