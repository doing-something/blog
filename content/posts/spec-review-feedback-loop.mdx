현재 회사에서 OpenSpec을 사용하여 기능 개발 자체뿐 아니라, 스펙을 보고 구현할 때 나중에 다시 흔들리지 않게 하는 개발 프로세스를 같이 다듬고 있다.

## 내가 사용하는 기본 사이클

```
explore → propose → validate → spec-review → apply → verify → archive
(탐색)     (제안)     (형식검증)     (내용검증)     (구현)   (검증)   (완료)
```

여기서 spec-review는 이 스펙대로 구현을 시작해도 괜찮은지 내용을 검토한다.
스펙이 모호하거나 빠진 게 있으면, 구현 도중에 에이전트가 알아서 판단하거나 멈추는 일을 막는 게 목적이다.

다음 5가지를 본다.

1. 완결성: 구현에 필요한 정보가 다 적혀 있는가?
2. 결정성: 애매한 부분 없이 하나로 결정돼 있는가?
3. 우회 가능성: 에이전트가 스펙을 무시하고 다른 길로 갈 여지가 있는가?
4. 응집도: 한 change 안의 내용이 서로 관련 있는 하나의 덩어리인가?
5. 분리 필요성: 너무 큰 덩어리라서 쪼개야 하는 건 아닌가?

검토 결과는 이렇게 나온다.

- Blockers — 이거 안 고치면 apply 못 함
- Warnings — 구현은 되지만 나중에 정리 필요
- Questions For Developer — 사람이 결정해줘야 하는 것
- Ready For Apply — yes/no

Blocker나 Question이 있으면 사람이 답하거나 스펙을 고친 뒤 다시 review를 돌린다.
통과하면 apply로 넘어간다.

## 문제

spec-review를 통과했는데도 구현 중에 문제가 뒤늦게 발견되는 경우가 있다.

예를 들어
- "이 데이터를 어디서 가져와야 하지?" — 스펙에 데이터 출처가 안 적혀 있었음
- "이 로직은 A 컴포넌트가 담당? B 컴포넌트가 담당?" — 책임 경계가 모호했음

이런 이슈가 apply(구현) 도중 또는 verify(검증) 때 발견되면, 그때서야 스펙을 고치거나 결정을 내려야 해서 진행이 멈추는 문제가 있었다. 그리고 이런 실패가 반복되어도 학습하는 구조가 없었다.

그래서 세 가지 기능을 추가했다.

## 1. spec-review에 실용적 확인 항목 추가

### 데이터 source/path

데이터를 어디서 가져오는지 스펙에 명시돼 있는가?를 미리 확인하게 한다.

- 어디서? → 어떤 API를 호출하는지 (예: GET /assessments/{id}/report)
- 어떻게? → 응답의 어떤 필드를 쓰는지 (예: response.data.testResults[].score)

이게 스펙에 안 적혀 있으면, 구현하는 에이전트가 직접 API 문서를 뒤지거나 추측해서 결정하게 된다.

### 구현 책임 경계

"이 로직을 누가 소유하는지 스펙에 명확한가?"

예를 들어 "로그인 상태가 아니면 로그인 페이지로 보낸다"라는 요구사항이 있다고 하면:

- 라우터 가드에서 처리할 수도 있고
- 각 페이지 컴포넌트에서 처리할 수도 있고
- API 호출 래퍼에서 401 받고 처리할 수도 있고

스펙에 안 정해져 있으면 에이전트가 하나를 골라야 한다.
이런 경계가 모호한 게 있으면 apply 전에 잡는 게 목적이다.

그리고 spec-review 결과를 항상 spec-review-report.md로 남기게 했다. 나중에 왜 통과했는지, 무슨 warning이 있었는지 회고할 수 있게 하려는 목적이다.

## 2. Feedback Tag 시스템

apply나 verify 중 뒤늦게 발견된 이슈를 4가지 태그로 분류한다.

- spec-review miss
spec-review가 잡았어야 하는데 놓친 것.
예: 모호한 조건이 review를 통과함
- verification gap
테스트/검증이 부족했던 것.
예: 시나리오는 있지만 테스트가 없었음
- new requirement
원래 제안 범위 밖의 새 요구사항
예: 구현 중 "이것도 필요하네" 발견
- implementation-only discovery
코드를 짜봐야만 알 수 있었던 것
예: 라이브러리 제약, 성능 이슈 등

이 태그는 tasks.md의 Feedback 섹션에 기록한다.

## 3. retro-spec-review — 회고 커맨드

아카이브된 최근 change 5개를 모아서 "spec-review가 잘 작동하고 있는가?"를 회고하는 커맨드이다.

하는 일은 다음과 같다.

- 최근 5개 change의 feedback tag를 모아서 분포를 봄
- 같은 태그가 2회 이상 반복되면 → "review 기준을 보강해야 할 수 있다"고 알려줌
- spec-review miss가 반복되면 → review 질문 강화 검토
- verification gap이 반복되면 → 테스트 규칙 보강 검토

자동화는 아직 계획에 없다. 작은 규모의 혼자하는 프로젝트라 자유롭게 실험하면서 개선하는 게 우선이다.

## 세 가지 개선을 적용한 뒤 첫 결과

개선된 spec-review를 단순한 업무에 돌려보았다.  
Ready For Apply: no — 즉 "이 스펙으로 구현 시작하면 안 된다"가 나왔다.

잡아낸 문제는 다음과 같다.

1. 라우트 경로가 두 가지로 섞여 있었음
스펙 안에서 같은 페이지의 URL이 /_authenticated/assessment-report와 /assessment-report로
혼용되고 있었다. 구현자가 둘 중 뭘 써야 하는지 알 수 없는 상태였다.
2. 날짜 표시 형식이 안 정해져 있었음
헤더에 reportDate를 보여줘야 하는데, 2026-04-10인지 2026.04.10인지 4월 10일인지 안 적혀 있었다. 구현자가 알아서 골라야 하는 상태였다.
3. 배너 문장 조합 규칙이 모호했음
노란 배너에 들어갈 문장을 여러 조건에 따라 조합해야 하는데, 그 규칙이 하나로 닫혀 있지 않았다. 같은 스펙을 읽고도 서로 다른 결과물이 나올 수 있는 상태였다. 

세 가지 모두 스펙이 틀린 건 아닌데, 하나로 결정이 안 돼 있어서 구현자마다 다르게 만들 수 있는 문제이다.
이전이었다면 이런 것들이 구현 도중에 발견됐을 텐데, spec-review 단계에서 미리 잡아서 스펙을 고친 뒤 구현에 들어갈 수 있게 된 것이다.

## 마무리

이 방식으로 일을 하면서 시행착오를 거치다 보니 요령이 좀 생기는 것 같다. 스펙을 중간에 고치는 일도 요즘엔 거의 없고, 구현까지 순조롭게 진행되고 있다.

초반에 새로 프로젝트 구성하면서 한꺼번에 여러가지를 처리하려는 욕심에 주먹구구식으로 일했던 것이 문제였다.

AI 없이 혼자 일할 때는 같은 상황에서도 사람이 정돈하면서 앞으로 나아갈 수 있으니까 별 문제가 없었던 것 같은데, AI를 관리해야 하는 내가 중심을 못잡고 있으면 내가 알 수 없는 상황에 휩쓸리게 된다. 이 경험이 너무 별로여서 정신이 바짝 들었다. 

spec-review라는 검증 도구는 전부터 만들어서 쓰고 있었다.
이번에 한 건, 그 도구가 제대로 작동하는지 데이터를 모으고(feedback tag), 주기적으로 돌아보고 보강하는(retro) 장치를 붙인 것이다.

워크플로우 도구 자체를 자기 점검하는 장치를 만든 건 이번이 처음이다. 돌려보면서 개선하고, 다른 단계에도 검증 장치를 추가하고자 한다.  

다음으로 하고 싶은 건 스펙 작성 단계에 좀 더 적극적으로 끼어드는 것이다.  
현재는 propose 단계에서 LLM이 스펙을 작성하면 내가 읽으면서 검증한다. 그런데 스펙 문서가 길어지면 집중력이 떨어져서 대충 넘어갈 때가 있고, 그러면 꼭 문제가 생기더라.
완성된 결과물을 사후 검증하는 것보다, 작성 과정에서 같이 다듬는 구조를 만들고 싶다.
