AI 코딩 에이전트로 개발하면서 혼란스러웠다.
주변에 물어볼 사람도 없었고, 온라인에서 나름 찾아보긴 했는데 뭐가 맞는 건지 잘 모르겠었다.
어찌되었든 회사에서 할 일은 있으니 그걸 하면서 작은 시도들을 해보았다.

1. Cursor, Claude Code로 대화형 개발을 했다.
2. 대화가 길어져서 새 세션마다 지시를 다시 해야 했다. docs/ 폴더에 프롬프트 문서를 마크다운으로 정리하고, 이를 기반으로 문서 기반 개발을 했다.
3. 문서를 기반으로 AI와 개발하다 보니, AI가 점점 스스로 작업을 판단하고 처리하려고 했다. 편하긴 했지만 최종 통제는 내가 가져야 한다고 봤다.
4. 작업의 범위, 절차, 산출물을 명확하게 통제할 수 있는 더 구조적인 시스템이 필요했다.

이렇게해서 찾은 게 스펙 주도 개발이었다.
그중 OpenSpec을 선택한 이유는 구조가 단순하고, 적용이 쉽고, 필요에 따라 유연하게 확장할 수 있기 때문이다.
이 유연성으로 인해 실제 작업 과정에서 발생하는 문제를 워크플로에 반영하기가 쉬웠다.
AI 에이전트와 개발하면서 겪은 문제를 기준으로, 에이전트가 임의 판단하지 못하도록 범위, 절차, 검증, 회고를 단계별로 고정하는 시스템으로 발전시켰다.

## 경험이 쌓인 뒤에 조금은 보이기 시작한다

AI 개발과 관련된 새로운 흐름과 시도에 대한 얘기가 SNS에 오고가는 걸 보면 불안했다.
하지만 나 스스로도 AI를 어떻게 다뤄야 하는지 아직 분명히 정리하지 못한 상태였고, 그런 상황에서 다른 방법론이나 기술을 추가로 익힌다고 잘 할 수 있을 것 같지가 않았다.
그래서 우선은 내가 맡은 업무를 해결하면서 앞서 말한 방식들을 실제 작업 속에서 먼저 해보고 맞지 않는 부분은 고쳐 나갔고, 그 과정은 이 블로그에 계속 남겨 왔다.
이제는 무엇을 할 때 어떤 나쁜 결과가 나오는지, 현재 무엇이 부족한지, 그리고 앞으로 어디를 개선해야 하는지가 조금씩 보이는 것 같다.  

사람들이 이야기하는 주제들을 나름의 기준을 가지고 살펴볼 수 있겠다는 생각이 들어서, 
[하네스 엔지니어링: 에이전트 우선 세계에서 Codex 활용하기](https://openai.com/ko-KR/index/harness-engineering/) 이 글을 이제야 읽었다.
올해 2월에 나온 글이니 늦게 본 편이지만, 아마 그때 읽었어도 제대로 이해하지는 못했을 것 같다.
그런데 앞의 경험을 거치고 나서 읽으니, 내가 겪었던 문제 및 그걸 해결하려고 했던 시도가 비슷해서 쉽게 읽혔다. 문제를 해결하는 과정에서 Claude Code, Codex를 활용했으니 LLM 개발사의 방향에 영향을 받아 비슷한 방향으로 움직이게 되었을 수도 있을 것이다. 그렇다고 해도 시간과 노력을 들여서 내가 헤쳐나간 것이기도 하다. 헤메는 것 같아도 어떻게든 문제를 해결하려고 매달리다 보면, 느리더라도 길을 찾게 되는 것 같다.

## 스펙 주도 개발 워크플로우에서 이미 하고 있는 것: Harness Engineering과의 접점

### 1. 리포지터리가 기록 시스템

설계 결정, 미해결 이슈, 작업 이력이 리포 안 마크다운 아티팩트로 관리된다.

- openspec/changes/ — change별 proposal, design, spec, tasks
- openspec/specs/ — archive 후 누적되는 main spec
- docs/rules/ — 테스트·컴포넌트 상세 규칙

### 2. AGENTS.md는 백과사전이 아니라 목차이다

핵심 규칙과 라우팅만 두고, 상세 규칙은 `docs/rules/*`로 분리한다. 
에이전트는 변경 대상 파일 유형에 따라 필요한 문서만 선택적으로 읽는다.

### 3. 계획은 일급 아티팩트

복잡한 작업의 목적, 결정, 검증 계획이 리포지터리 안 실행 아티팩트로 남는다.
OpenSpec의 proposal.md → design.md → tasks.md가 그 역할을 한다.
특히 `design.md`의 `Decisions`, `GRAY-AREAS → Decisions` 승격, `tasks.md`의 `Feedback`은 의사 결정과 수정 이력을 기록하는 구조다.

### 4. 사람이 조정하고, 에이전트가 수행한다

사용자는 `opsx:*` 워크플로우를 통해 의도를 입력하고, 에이전트는 그에 따라 아티팩트를 생성하고 구현과 검증을 수행한다.

### 5. 문서 아티팩트도 기계적으로 검증한다

문서를 사람이 읽고 판단하는 것에 맡기지 않고, 형식·구조·정합성을 기계적으로 검사한다.
우리 프로젝트에서는 `opsx:validate`가 change 아티팩트의 형식, 참조, 추적성을 검증한다.
OpenAI 글도 구조화된 기록 시스템이 기계적 점검에 적합하다고 본다.

### 6. 에이전트가 어려움을 겪으면 환경을 고친다

에이전트가 반복해서 막히는 지점은 개인 숙련도의 문제가 아니라 시스템 설계의 문제로 보고, 문서와 워크플로우를 고친다.
우리 프로젝트도 규칙 문서 추가, 워크플로우 단계 분리, 어댑터 정렬, 회고 기록 추가 같은 방식으로 환경을 계속 보강해 왔다.

### 7. 에이전트가 만든 어긋남을 추적하고 정리한다

에이전트가 일하다 보면 스펙과 구현이 어긋나거나, 검증이 빠지거나, 이전 리뷰에서 못 잡은 것들이 쌓인다. 이걸 그냥 두지 않고 기록해서 반복되면 규칙을 고친다.

- retro-spec-review — 최근 archive를 회고하여 반복 패턴 발견
- tasks.md Feedback 섹션 — spec-review miss, verification gap 등 태그로 분류
- "같은 tag 2회 이상 반복될 때만 기준 보강 검토" — 과잉 반응 방지 


## OpenAI Harness Engineering의 방식과의 차이점

### 1. 에이전트 자율성 수준

OpenAI는 에이전트가 훨씬 더 많은 일을 스스로 한다. 코드 작성, 리뷰, PR 생성과 병합까지 폭넓게 맡긴다.
우리 프로젝트는 그보다 개발자의 확인과 통제가 더 강하다.

### 2. 에이전트가 앱을 직접 돌려보는가

OpenAI는 에이전트가 앱을 직접 실행하고, 브라우저를 조작하고, 로그와 메트릭까지 보면서 문제를 찾고 고친다.
우리 프로젝트는 문서와 절차 쪽은 갖춰져 있지만, 그런 실행 환경은 부족하다. 

### 3. 장기적인 운영 상태를 관리하는가

OpenAI는 장기 계획, 품질 점수, 기술 부채 같은 운영 정보를 따로 계속 관리한다.
우리 프로젝트는 change 하나하나의 기록은 잘 남기지만, 프로젝트 전체의 건강 상태를 추적하는 구조는 아직 없다.

### 4. 빠르게 vs 정확하게

OpenAI는 빨리 병합하고 문제가 생기면 바로 고치는 쪽이다.
우리 프로젝트는 병합 전에 스펙과 어긋나지 않는지 먼저 확인하는 쪽이다.

### 5. 지속적인 정리 자동화

OpenAI는 백그라운드 에이전트가 돌면서 문서가 낡았는지, 코드가 규칙에서 벗어났는지 계속 점검하고 PR을 올린다.
우리 프로젝트는 회고는 하지만, 상시 돌아가는 자동 정리는 아직 없다.

### 6. 규칙을 도구로 막는가

OpenAI는 아키텍처 규칙을 어기면 빌드가 깨지게 만들어놨다.
우리 프로젝트는 규칙이 문서에 있어서, 에이전트가 읽고 따르길 기대하는 형태다. 안 지켜도 빌드는 통과한다.


지금 OpenAI처럼 자율성을 높이는 건 아직 이르다.
우리는 단계별 확인과 검증을 더 탄탄하게 만드는 게 먼저다.

## 앞으로 할 일 

지금까지는 AI가 따를 규칙과 절차를 만드는 단계였고, 지금 부족한 건 실행 결과를 증거로 남기고, 검증을 일관되게 하고, 반복 작업을 자동화하는 쪽이다

### 1. 검증 근거 기록 방식 통일

현재 report에는 판정과 근거가 들어가지만, 근거를 남기는 형식이 일정하지 않다.
각 판정마다 "무엇을 보고 이렇게 판단했는지"를 같은 방식으로 남기는 규칙을 정하는 것이 좋겠다.

### 2. Playwright 검증 하네스 도입

지금 verify는 코드를 읽고 "이 코드가 스펙대로 된 것 같다"고 판정한다.
실제로 브라우저에서 돌려본 게 아니다.
실제 브라우저에서 돌려보고 맞는지 확인하는 자동화된 브라우저 테스트를 붙인다.

### 3. opsx runner 설계 및 구현

 `apply`, `verify`, `spec-review`는 목적은 다르지만, 매번 반복하는 공통 작업이 있다.
- 관련 spec 파일 읽기
- `tasks.md`와 `validation` 항목 대조하기
- report 초안에 들어갈 기본 정보 모으기

이 공통 작업을 한 곳에 모아서 코드로 만들면 좋겠다.
지금은 이 절차가 command 텍스트(.md 파일)에 글로 적혀 있어서, AI가 읽고 따라하는 방식이다. 그래서 실행할 때마다 해석 차이가 조금씩 생길 수 있다.
`opsx runner`를 만들면, 스크립트 한 번 실행으로 이런 반복 작업을 같은 방식으로 처리할 수 있다.
매번 똑같이 해야 하는 수집·대조·정리 작업만 코드로 고정하고, 해석과 판단은 계속 AI가 맡는 구조다.

### 4. Scenario ID와 실제 검증 수단의 연결 강화

현재도 `spec.md`의 Scenario ID는 `tasks.md` 검증 항목과 연결되고 `verify-report.md`에서도 시나리오 단위로 판정한다.  즉 문서 기준의 추적 자체는 이미 하고 있다. 다음은 이 연결을 실제 검증 수단까지 더 분명하게 이어 주는 것이 필요하다.

즉 각 Scenario ID마다 최소 하나는 더 명확하게 연결되게 만든다.
- 테스트
- Playwright flow
- 스크린샷
- 수동 검증 항목

이렇게 하면 verify가 문서 비교와 코드 읽기에만 머무르지 않고, 실제 실행 결과를 근거로 삼기 쉬워진다.
핵심은 "이 시나리오는 무엇으로 확인할 것인가"를 지금보다 더 분명하게 남기는 것이다. 그래야 검증이 사람이나 에이전트의 해석에 덜 의존하게 된다.

### 5. 자주 해석이 갈리는 영역에 짧은 맵 문서 추가

`AGENTS.md`를 키우기보다, 자주 해석이 갈리는 feature에만 짧은 맵 문서를 두는 것이 좋겠다.
이 문서에는 긴 설명 대신
- 상태 소유권
- API 경계
- 금지 패턴
정도만 적는다.

## 그 외 할 일

이제는 AI 에이전트를 만들고 싶다. 아직 회사에서 이런 일을 내게 아직 맡기진 않아서 뭘 해볼수 있을까 생각해봤다.  
중환자실 재활 운동 앱에서 매번 아이패드와 실사용자 동작에 의존하는 개발·테스트 비용이 너무 크다고 생각했다.
최종 검증은 실제 기기에서 하되, 사람과 아이패드에 의존하던 개발·테스트를 agent가 재현 가능하게 돌리도록 바꾼 뒤, 
실제 수행과의 오차를 줄이는 방향으로 발전시킬 수 있지 않을까 해서 이걸 해봤으면 좋겠다.
